# uTools 插件开发

以当前官方文档为唯一事实依据，构建和审查 uTools 插件。确保每项实现决策都能追溯到本次任务中已核验的文档页面。

## 必须执行的工作流

1. 修改前检查现有项目。识别项目框架、构建输出、`plugin.json`、入口页面、preload 脚本、包管理器以及用户已有的修改。
2. 将需求映射到涉及的 uTools 领域：配置、功能指令、生命周期事件、窗口/UI、系统访问、存储、浏览器自动化、支付、AI、MCP、Sharp、FFmpeg、打包或发布。
3. 阅读 `references/official-docs.md`，然后在当前任务中打开 `https://www.u-tools.cn/docs/developer/` 的相关页面。直接 URL 未知时，通过侧边栏链接或官方搜索结果查找。
4. 核验实现所使用的每个 uTools 专属 API 名称、签名、返回值、回调/事件时机、配置字段、平台限制和发布要求。官方页面可访问时，不要依赖记忆。
5. 实现满足需求的最小完整改动。保留用户不相关的修改，并遵循仓库现有约定。
6. 运行项目相关的 lint、类型检查、测试和构建。当打包行为相关时，检查编译输出。
7. 按下方 uTools 集成检查清单进行审查。报告查阅过的官方页面、重要限制以及任何未能完成的验证。

如果官方文档无法访问，只能在明确标注假设的情况下继续。不要声称某个 API 或规则是最新的。

## 信息源规则

- 将当前 uTools 官方开发者文档视为 uTools 行为的权威依据。
- 优先查阅准确的 API 或配置页面，而不是入门页面、示例、博客、代码仓库或记忆。
- 仅在官方文档无法解答工具链或生态问题时使用非官方来源，并单独标注。
- 区分文档事实与实现推断。如果示例与其上下文规范冲突，说明冲突，并遵循明确的规范，除非运行时测试证明并非如此。
- 在最终回复中记录页面 URL 和访问日期。不要将整套文档复制到此 Skill 中。

## 架构边界

- 将普通展示逻辑和浏览器安全的应用逻辑保留在前端。
- 将 Node.js 或 Electron 访问能力放入已配置的 preload 脚本，并向前端暴露范围有限的 API。
- 当面向具体任务的函数已足够时，绝不能通过 `window` 暴露不受限制的文件系统、Shell、进程或凭据访问能力。
- 在 preload 代码中校验前端传入的路径、命令、URL 和其他高权限输入。
- 不要将密钥写入源代码、`plugin.json`、日志或打包资源。
- 添加 Node 依赖前，通过当前的 preload 和目录结构文档核验 CommonJS、依赖位置、源码可读性和打包规则。

## 集成检查清单

### `plugin.json`

- 根据当前官方配置页面校验 JSON 语法和所有使用的字段。
- 确认引用的 `logo`、`main`、`preload` 及其他路径相对于打包后的插件目录真实存在。
- 检查每个 `features` 条目、唯一的 `code`、指令形式、说明、平台规则和动态功能要求。
- 防止仅用于开发的 URL 或配置进入生产输出。

### Preload 与高权限能力

- 确认打包目录中包含 preload 文件。
- 核验模块格式、运行时兼容性、依赖位置以及审查/源码可读性要求。
- 暴露小型且有明确命名的前端接口；避免泄露原始 Node 或 Electron 模块。
- 在权限边界处理错误，并返回可序列化且有明确约定的值。

### 前端与构建输出

- 确保交付给 uTools 的是可部署的 HTML、CSS 和 JavaScript，而不是未经编译的框架源码。
- 确认资源路径可以从打包后的入口页面正常访问。
- 使用 Vite 或其他开发服务器时，核验当前 `development.main` 指引，并单独测试构建输出。
- 注意 preload 改动可能需要彻底终止并重新打开插件；从当前调试文档核验这一点。

### 运行时行为

- 测试每个功能指令以及相关的进入/退出生命周期行为。
- 测试空值、格式错误、用户取消和权限拒绝等输入。
- 当插件声明跨平台支持时，测试受影响的操作系统。
- 使用本地数据库时，核验数据持久化冲突/版本修订行为。

### 打包与发布

- 仅打包预期的编译输出和必需且源码可读的 preload 依赖。
- 检查版本、图标、截图、使用说明、隐私/安全影响以及当前审核要求。
- 除非当前官方文档另有说明，否则将离线安装包视为测试或内部分发方式。

## 回复格式

执行实现任务时，总结：

- 修改内容及其验证方式；
- 查阅过的 uTools 官方页面及访问日期；
- 相关的平台、安全、打包或 API 限制；
- 剩余假设或需要在 uTools 开发者工具中手动检查的项目。

对于仅需解释的问题，直接回答，并引用查阅过的准确官方页面。
